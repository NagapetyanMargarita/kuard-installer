# 🛡️ Kuard-installer
Минималистичный инсталлятор для развёртывания базовых security-сервисов в Kubernetes-кластере.

## Компоненты

| Сервис | Версия | Назначение |
|------------|------------|------------|
| [vault](https://github.com/hashicorp/vault-helm/tree/main) | 0.32.0 | Хранение и управление секретами (токены, пароли, сертификаты) с контролем доступа |
| [minio](https://github.com/minio/minio/tree/master/helm/minio) | 5.4.0 | S3-совместимое объектное хранилище (используется как бекенд для Loki) |
| [fluent-bit](https://github.com/fluent/helm-charts) | 0.56.0 | Легковесный сборщик логов с узлов Kubernetes |
| [loki](https://github.com/grafana/loki/tree/main/production/helm/loki) | 6.28.0 | Агрегация и хранение логов (как Prometheus, но для логов) |
| [kube-prometheus-stack](https://github.com/prometheus-community/helm-charts/tree/main/charts/kube-prometheus-stack) | 80.10.0 | Полный мониторинг кластера: Prometheus + Grafana + Alertmanager
| [falco](https://github.com/falcosecurity/charts/tree/master/charts/falco) | 8.0.1 | Обнаружение аномалий и угроз в реальном времени (runtime security)
| [trivy-operator](https://github.com/aquasecurity/trivy-operator/tree/main/deploy/helm) | 0.32.1 | Автоматическое сканирование образов и запущенных подов на уязвимости внутри кластера

* RAM > 4
* CPU > 2

## ✨ Краткое описание

Kuard-installer был создан как комплексное решение, которое позволяет развернуть базовый security-стек в Kubernetes-кластере без ручного перебора десятков Helm-команд и долгой настройки каждого компонента. Всё, что требуется от пользователя, - заполнить конфигурационный файл my_vars.yml, указав нужные параметры, и запустить установочный скрипт run.sh.

При запуске автоматически создаётся изолированное виртуальное окружение Python, в которое подтягиваются все зависимости из requirements.txt, а также загружаются необходимые Ansible-коллекции (`kubernetes.core`,`community.hashi_vault`). Пароль для доступа к сервисам вводится интерактивно и не отображается в процессе ввода.

В качестве дополнительного функционала было добавлено:
* Логирование: При каждом запуске инсталлятора в папке _logs создается файл в формате `<ДАТА>_installer.log`, который в последствии можно использовать для отлаживания ошибок.
* Доп проверки: В лоюой момент времени пользователь может запустить скрипт с ключом `./run.sh --check`, который запустит код проверки состояния кластера, работоспособности vault.

Исталлятор состоит из следующих шагов, каждому из которых соответсвует собственный playbook и role: 
1. Pre_checks: Начальные проверки, отслеживающие наличие обязательных инструментов (helm, kubectl и т.п.) и добавляющие репозитории устанавливаемых сервисов.

2. Vault:  В процессе настройки создаются namespace `infra` и разворачивается vault с двумя типами доступа: `infra-ro` (только чтение секретов) и `infra-rw` (чтение и запись). Доступ к ним привязан к Service Accounts: `infra-vault-ro` получает права на чтение, а `infra-vault-rw` - на чтение и запись через метод аутентификации kubernetes. Это позволяет приложениям в кластере безопасно взаимодействовать с Vault, используя свой Service Account, и получать ровно те права, которые им нужны.

3. MinIO: S3-совместимое объектное хранилище, развертываемое в режиме `standalone` с одной репликой в namespace `monitoring` (как и сервисы из 4-5 пунктов). В проекте используется в качестве бэкенда для Loki: именно сюда будут сохраняться чанки (блоки сжатых логов) и индексы. Создаются три бакета: `loki-chunks` для хранения самих логов, `loki-ruler` для правил агрегации и `loki-admin` для служебных данных. Доступ к MinIO осуществляется по внутреннему адресу `minio.logs.svc.cluster.local:9000` с использованием стандартных учетных данных `minio/minio123`. Хранилище настроено на сохранение данных на `persistent volume` размером 5 ГБ с использованием storage class standard. Консоль управления MinIO отключена, так как сервис используется только как внутреннее хранилище.

4. Fluent Bit: легковесный сборщик логов, работающий как DaemonSet. Fluent Bit читает логи контейнеров из файлов `/var/log/containers/*.log`, используя парсеры для форматов docker и cri, а также логи kubelet через systemd. Логи обогащаются метаданными Kubernetes: извлекаются namespace, pod name, container name, которые добавляются в виде меток (labels) при отправке. Фильтры настроены таким образом, чтобы корректно обрабатывать многострочные сообщения и исключать ненужные поля из kubernetes. Выходной поток направляется в Loki по адресу `loki.logs.svc.cluster.local:3100`, данные сжимаются gzip для уменьшения трафика. Fluent Bit также предоставляет метрики Prometheus на порту 2020, включая эндпоинт `/api/v1/metrics/prometheus`. RBAC настроен на чтение метаданных подов, но без доступа к событиям кластера и без привилегий nodeAccess.

5. Loki: система агрегации и хранения логов, работающая в режиме `SingleBinary`. Такая конфигурация выбрана для упрощения развертывания в небольших средах. Loki настроена на хранение данных в MinIO через S3-совместимый API: чанки сохраняются в бакет `loki-chunks`, данные ruler - в `loki-ruler`, административные данные - в `loki-admin`. Для работы с MinIO используется подключение по HTTP без TLS. Включена аутентификация, поддержка структурированных метаданных, работает ruler API для генерации алертов на основе логов. Ресурсы пода ограничены: 200m CPU и 512 Mi запрос, до 500m CPU и 2 Gi лимит, данные также сохраняются на PVC размером 5 ГБ.

6. Falco: система обнаружения угроз и аномалий в реальном времени, развертывается как DaemonSet в namespace `infra`. В проекте настроена на логирование подозрительных событий в формате JSON с указанием временной метки ISO 8601 и минимальным уровнем важности `warning`. Все события также отправляются в Loki через компонент Falcosidekick, который принимает алерты Falco и форвардит их в Loki по адресу `http://loki.logs.svc.cluster.local:3100`. Для мониторинга самого Falco включены метрики Prometheus: они собираются на порту 8765, ServiceMonitor уже настроен с интервалом опроса 30 секунд и добавлены соответствующие аннотации для автоматического обнаружения. Драйвер Falco работает в автоматическом режиме, обеспечивая перехват системных вызовов на уровне ядра. Сборщики включены, в частности для отслеживания контейнерного движка. Ресурсы ограничены: запрос 100m CPU и 256 Mi памяти, лимиты 500m CPU и 512 Mi. Falco отслеживает такие события, как неожиданные сетевые соединения, запуск привилегированных контейнеров, монтирование чувствительных путей, выполнение шелл-команд внутри подов и другие аномалии.

7. Trivy-Operator: автоматический сканер безопасности для Kubernetes, работающий по модели operator. В проекте настроен на сканирование образов на уязвимости и поиск секретов (токенов, паролей, ключей) в этих образах. Сканированию подлежат следующие типы ресурсов: pod, replicaset, statefulset, daemonset, cronjob и job. Из проверок исключены системные namespace: `kube-system`, `trivy-system` и `monitoring`, а целевых namespace не задано - значит, проверяются все остальные. Конфигурационное сканирование, анализ RBAC и инфраструктуры отключены, фокус только на безопасности образов. Результаты сканирования хранятся в отчётах (reports) в течение 12 часов, а сами Jobs сканирования удаляются через час. Включена генерация метрик для Prometheus - они собираются ServiceMonitor, настроенным для работы с мониторингом в namespace `monitoring`. Trivy-Operator автоматически обнаруживает новые образы, запускает сканирование и публикует отчёты, которые можно просматривать через kubectl или визуализировать в Grafana на основе собранных метрик.

8. Prometheus: основная система сбора метрик, входящая в состав `kube-prometheus-stack`. В проекте настроен на сбор метрик со всех компонентов Kubernetes: kubelet, API server, etcd, controller-manager, scheduler, а также с node-exporter (метрики узлов), kube-state-metrics (состояние объектов кластера) и любых сервисов, аннотированных для Prometheus. Метрики собираются с интервалом 30 секунд. Prometheus доступен внутри кластера по адресу `http://monitoring-kube-prometheus-prometheus.monitoring:9090/`, который используется как источник данных для Grafana. Правила оповещения (alerting rules) настроены на отправку алертов в Alertmanager для дальнейшей маршрутизации по email. Включены стандартные алерты: недоступность узлов, высокое потребление CPU/памяти, проблемы с подами (CrashLoopBackOff, длительное ожидание), заполнение диска, сбои компонентов control plane. Prometheus работает в режиме `HA`, но поддерживает возможность масштабирования.

9. Grafana: платформа для визуализации метрик и логов, развернутая в составе мониторингового стека. Доступ к Grafana организован через NodePort на порту 30000 с учетными данными `admin/admin`. В качестве источников данных настроены два датасорса: Prometheus (основной, используется по умолчанию) для получения метрик и Loki для запросов логов с ограничением в 1000 строк на ответ. Импортированы готовые дашборды: для отображения состояния Kubernetes-кластера (gnetId 7249), мониторинга kube-state-metrics (gnetId 13332), анализа использования томов (gnetId 11454) и дашборд Trivy Operator для отображения уязвимостей и секретов в образах (gnetId 17813). Также добавлен пользовательский дашборд Node Exporter, загружаемый из локального файла. Все дашборды привязаны к датасорсу Prometheus. Автоматическое создание стандартных дашбордов и датасорсов отключено, чтобы использовать только предварительно настроенные источники.

10. Alertmanager: компонент, отвечающий за маршрутизацию и обработку оповещений от Prometheus. В проекте настроена отправка уведомлений через SMTP: используется указанный почтовый сервер, технический ящик отправителя и пароль приложения. Алерты группируются по имени, namespace и критичности с интервалом ожидания 30 секунд перед отправкой. Повторные уведомления отправляются каждые 4 часа, если проблема не решена, а интервал между групповыми отправками составляет 5 минут. Единственный активный получатель - email-адрес, на который отправляются все алерты, включая уведомления о восстановлении (send_resolved: true). Стандартный алерт Watchdog (проверка работоспособности самого Alertmanager) отправляется в null-приемник и не рассылается по почте. Для создания отдельной почти для инсталлятора на Google можете воспользоваться следующей официальной [инструкцией](https://support.google.com/mail/answer/185833?hl=ru). 

После успешной установки будут доступны веб-интерфейсы: Grafana на порту 3000 с предустановленными дашбордами для метрик кластера, логов Loki, событий Falco и отчётов Trivy Operator; Vault UI на порту 8200 для управления секретами; а также MinIO Console на порту 9001. Вы также можете добавлять в Grafana свои собственные дашборды. 

Если необходимо удалить все развёрнутые сервисы, достаточно запустить скрипт с ключом `--remove | -r` 

## 🚀 Запуск
1. Подставьте собственные значения в файл my_vars.env.
2. Запустите скрипт
```
./run.sh
```
3. Дождитесь завершения и перейдите в Grafana с default кредами admin/admin (впоследствии можно поменять)

## 🗑️ Удаление
Запустите следующий скрипт, позволяющий удалить все установленные чарты с namespace:

```
./run.sh --remove
```
> Можно использовать сокращенную версию ключа `-r`.
